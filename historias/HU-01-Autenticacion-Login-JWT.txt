HU-01 - Autenticación de usuario con JWT

Como usuario registrado,
quiero iniciar sesión con mi correo y contraseña,
para obtener un token JWT con mis permisos de negocio que me permita consumir los servicios protegidos del sistema.

## Historia de Usuario Enriquecida

### Contexto y Objetivo de Negocio
El sistema debe permitir que los usuarios registrados accedan de forma segura a los servicios protegidos mediante autenticación basada en JWT. El token debe incluir los permisos de negocio del usuario (REPORT_READ, REPORT_DOWNLOAD) para que los servicios puedan autorizar acciones específicas sin consultarlos nuevamente.

### Descripción Funcional Detallada
El usuario debe introducir su correo y contraseña en el endpoint de autenticación. El sistema debe validar las credenciales, comprobar que el usuario exista y esté activo, verificar la contraseña contra su hash almacenado. Si todo es correcto, debe consultar los permisos del usuario desde el servicio de autorización (authorization-service) y emitir un JWT firmado incluyendo esos permisos en el claim `permissions`. Si las credenciales no son válidas, el sistema debe rechazar el acceso sin revelar si el fallo fue por correo o contraseña.

### Criterios de Aceptación
- [ ] El usuario puede autenticarse mediante `POST /api/auth/login` con correo y contraseña válidos.
- [ ] Si las credenciales son correctas y el usuario está activo, el sistema responde con HTTP 200 y un JWT válido.
- [ ] La respuesta incluye `tokenType`, `accessToken`, `expiresIn` y datos básicos del usuario.
- [ ] Si el correo no existe o la contraseña es incorrecta, el sistema responde con HTTP 401 y un mensaje genérico.
- [ ] Si el usuario está inactivo o bloqueado, no se genera token.
- [ ] El token incluye `subject`, permisos de negocio (permissions), fecha de emisión y expiración.
- [ ] La contraseña nunca se devuelve en la respuesta ni se expone en logs.

### Casos Límite y Validaciones
- Caso límite: correo o contraseña vacíos; el sistema debe rechazar la petición con error de validación.
- Caso límite: token expirado; los servicios protegidos deben rechazarlo.
- Caso límite: múltiples intentos fallidos; debería contemplarse control de abuso o bloqueo temporal si aplica.
- Validación: el usuario debe existir y estar activo antes de emitir el token.
- Validación: la contraseña debe compararse únicamente contra su hash BCrypt.

### Requisitos No Funcionales Relevantes
- Seguridad: uso de JWT firmado, almacenamiento seguro de contraseñas, protección contra exposición de información sensible.
- Rendimiento: autenticación con baja latencia y validación eficiente del token.
- UX / Accesibilidad: mensajes claros y consistentes para errores de autenticación.
- Observabilidad / Logging: registrar eventos de autenticación sin incluir credenciales ni tokens completos.
- Documentación API: configurar OpenAPI/Swagger para documentar este endpoint y sus contratos relacionados.

### Requisitos de Testing (alto nivel)
- Pruebas unitarias: validación de credenciales, comparación con BCrypt, generación y validación de JWT.
- Pruebas de integración: flujo completo de login contra la base de datos y el servicio de autenticación.
- Pruebas E2E: autenticación exitosa y acceso posterior a un recurso protegido.

### Consideraciones Técnicas
- Backend / API:
  - `POST /api/auth/login`
  - `AuthController`, `AuthService`, `JwtService`, `UserRepository`, `LoginRequest`, `LoginResponse`, `UserResponse`
- Seguridad:
  - `Spring Security`, `BCryptPasswordEncoder`, validación de token y expiración.
- Errores:
  - HTTP 400 para validación, HTTP 401 para credenciales inválidas.

Objetivo técnico:
Implementar un backend en Java 17 con Spring Boot que permita autenticar usuarios mediante credenciales válidas y generar un token JWT firmado.

Alcance funcional:
- Crear endpoint POST /api/auth/login.
- Validar correo y contraseña.
- Consultar el usuario en base de datos.
- Verificar contraseña usando BCrypt.
- Generar un JWT con claims mínimos.
- Retornar token, tipo de token, expiración y datos básicos del usuario.
- Rechazar credenciales inválidas sin revelar si falló el correo o la contraseña.

Buenas prácticas esperadas:
- Usar Java 17.
- Usar Spring Boot 3.x.
- Usar Spring Security.
- Usar BCryptPasswordEncoder para validar contraseñas.
- No guardar contraseñas en texto plano.
- No retornar información sensible del usuario.
- Manejar errores con una respuesta estándar.
- Configurar expiración del token.
- Mantener separación por capas: controller, service, repository, dto, entity, security.
- Usar DTOs para entrada y salida.
- Validar entrada con jakarta.validation.
- Documentar el contrato con OpenAPI/Swagger para no omitir campos, errores ni reglas del flujo.

Endpoint:
POST /api/auth/login

Content-Type: application/json

{

  "password": "Password123!"

Ejemplo de salida exitosa HTTP 200:

Reglas de negocio:
- El usuario debe existir y estar activo.
- La contraseña debe coincidir con el hash almacenado.
- Si el usuario está bloqueado o inactivo, no debe generarse token.
- El token debe incluir subject, permisos (consultados desde authorization-service), fecha de emisión y expiración.
- Los permisos se mapean a partir de los roles del usuario: ADMIN → [REPORT_DOWNLOAD, REPORT_READ], VIEWER → [REPORT_READ].
- Si authorization-service no responde, se debe rechazar el login.

Criterios de aceptación:
1. Dado un usuario activo con credenciales válidas, cuando invoque POST /api/auth/login, entonces el sistema debe responder HTTP 200 con un JWT válido.
2. Dado un usuario inexistente o contraseña incorrecta, cuando invoque POST /api/auth/login, entonces el sistema debe responder HTTP 401 con mensaje genérico.
3. Dado un usuario inactivo, cuando intente iniciar sesión, entonces el sistema debe responder HTTP 403 o 401 según la política definida.
4. El token generado debe poder validarse posteriormente en servicios protegidos.
5. La contraseña nunca debe aparecer en logs ni respuestas.
