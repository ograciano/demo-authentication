HU-03 - Consultar permisos de usuario dinámicamente (CONSUMO INTERNO)

Como servicio de autenticación,
quiero consultar internamente los permisos de un usuario específico desde authorization-service (vía Feign),
para emitir un JWT con los permisos actualizados sin hardcodearlos en la lógica de autenticación.

## Historia de Usuario Enriquecida

### Contexto y Objetivo de Negocio
Durante el login, el servicio de autenticación debe consultar dinámicamente los permisos de un usuario desde authorization-service. Este es un **consumo interno vía Feign Client** (no endpoint público). Esto desvincula la lógica de asignación de permisos de la autenticación, permitiendo cambios en permisos sin redeploy de autenticación.

### Descripción Funcional Detallada
Durante el login (POST /api/auth/login), después de validar credenciales, authentication-service invoca internamente a authorization-service (vía Feign) para obtener los permisos del usuario. La consulta debe ser rápida y debe retornar una lista de permisos de negocio (REPORT_READ, REPORT_DOWNLOAD, etc.). Si el usuario no existe o no tiene permisos, se retorna lista vacía. Si authorization-service no responde, se puede omitir la consulta (fallback a lista vacía) para no bloquear el login.

### Criterios de Aceptación
- [ ] authorization-service expone GET /api/permissions/users/{userId} (uso interno).
- [ ] La respuesta incluye un array de permisos (strings) asociados al usuario.
- [ ] Si el usuario no existe, se retorna lista vacía (no 404).
- [ ] Si el usuario no tiene permisos, se retorna lista vacía.
- [ ] La consulta es rápida (idealmente <100ms).
- [ ] authentication-service invoca este endpoint vía Feign Client en el login.
- [ ] En caso de fallo de authorization-service, se usa fallback (lista vacía, sin fallar login).

### Casos Límite y Validaciones
- Caso límite: usuario con ID inválido; retornar 404 o lista vacía.
- Caso límite: servicio de autorización no disponible; gestionar timeout y circuit breaker.
- Validación: el ID del usuario debe ser numérico válido.
- Validación: la lista de permisos debe contener solo valores válidos conocidos.

### Requisitos No Funcionales Relevantes
- Seguridad: solo servicios internos pueden consultar este endpoint (sin exposición pública).
- Rendimiento: caché opcional de permisos para reducir latencia.
- Confiabilidad: manejo de fallos de conectividad con reintentos.
- Observabilidad: registrar consultas de permisos con IDs de usuario (sin datos sensibles).
- Documentación API: configurar OpenAPI/Swagger y marcar este contrato como interno para evitar exposición pública accidental.

### Requisitos de Testing (alto nivel)
- Pruebas unitarias: validación de formato de respuesta, manejo de casos vacíos.
- Pruebas de integración: consulta contra base de datos de permisos.
- Pruebas de conectividad: simulación de timeout y error en authorization-service.

### Consideraciones Técnicas
- Backend / API:
  - `GET /api/permissions/users/{userId}`
  - PermissionController, PermissionService, UserPermissionRepository
  - Respuesta: `{"userId": 1, "permissions": ["REPORT_READ", "REPORT_DOWNLOAD"]}`
- Integración:
  - Feign Client desde authentication-service a authorization-service.
  - Configuración de URL sin Eureka (properties estática).
- Errores:
  - HTTP 200 con lista vacía si no hay permisos.
  - HTTP 404 si el usuario no existe (opcional).
  - HTTP 503 si authorization-service está caído.

Objetivo técnico:
Exponer endpoint en authorization-service que retorne permisos de un usuario por su ID, permitiendo consulta desde authentication-service vía Feign Client.

Alcance funcional:
- Crear endpoint GET /api/permissions/users/{userId}.
- Consultar permisos desde base de datos o caché.
- Retornar array de permisos de negocio.
- Manejar usuario no existente.
- Manejar fallos de conectividad.

Buenas prácticas esperadas:
- Usar DTO para respuesta.
- Usar validación de entrada.
- Usar transacciones de solo lectura.
- Implementar caché si aplica.
- Loguear consultas sin datos sensibles.
- Documentar endpoint con OpenAPI.
- Aclarar en la documentación que este endpoint se consume solo por Feign desde authentication-service.

Endpoint:
GET /api/permissions/users/{userId}

Headers:
Accept: application/json
X-Service-Auth: <token-interno> (opcional, si se asegura con token)

Ejemplo de salida exitosa HTTP 200:
{
  "userId": 1,
  "permissions": ["REPORT_READ", "REPORT_DOWNLOAD"],
  "timestamp": "2026-05-13T14:30:00"
}

Ejemplo sin permisos HTTP 200:
{
  "userId": 2,
  "permissions": [],
  "timestamp": "2026-05-13T14:30:00"
}

Ejemplo de usuario no encontrado HTTP 404:
{
  "timestamp": "2026-05-13T14:31:00",
  "status": 404,
  "error": "Not Found",
  "message": "Usuario no encontrado",
  "path": "/api/permissions/users/999"
}

Reglas de negocio:
- Solo servicios internos pueden consultar este endpoint.
- Los permisos son consultados de la base de datos de permisos.
- Un usuario puede tener cero o más permisos.
- Los permisos conocidos son: REPORT_READ, REPORT_DOWNLOAD.

Criterios de aceptación:
1. Dado un usuario con permisos asignados, cuando invoque GET /api/permissions/users/{id}, entonces debe retornar HTTP 200 con lista de permisos.
2. Dado un usuario sin permisos, cuando invoque GET /api/permissions/users/{id}, entonces debe retornar HTTP 200 con lista vacía.
3. Dado un ID de usuario inválido, cuando invoque GET /api/permissions/users/{id}, entonces debe retornar HTTP 404 o lista vacía según política.
4. El endpoint debe ser accesible solo desde servicios internos.
5. La respuesta no debe incluir información sensible del usuario más allá del ID.

Sugerencia de estructura técnica:
- PermissionController
- PermissionService
- PermissionRepository
- UserPermissionEntity
- PermissionResponseDTO
- GlobalExceptionHandler
