HU-04 - Asignar permiso predeterminado a nuevo usuario (CONSUMO INTERNO)

Como servicio de autenticación,
quiero asignar automáticamente el permiso predeterminado (REPORT_READ) a un usuario nuevo via authorization-service (vía Feign),
para que esté habilitado con permisos iniciales desde el momento de su registro.

## Historia de Usuario Enriquecida

### Contexto y Objetivo de Negocio
Durante el registro (POST /api/auth/register), después de crear el usuario, authentication-service debe invocar internamente authorization-service (vía Feign) para asignar el permiso inicial REPORT_READ. Este es un **consumo interno, no endpoint público**. Permisos adicionales (como REPORT_DOWNLOAD) serán asignados posteriormente por administradores a través de endpoints específicos en authorization-service.

### Descripción Funcional Detallada
Durante el registro (POST /api/auth/register), después de guardar el usuario con rol VIEWER, authentication-service invoca internamente authorization-service (vía Feign) para asignar el permiso REPORT_READ. Si la asignación falla, la transacción debe revertirse: se elimina el usuario creado y se retorna error (rollback atómico). Si es exitosa, el cliente recibe confirmación de registro completo. El proceso es idempotente: si se intenta asignar el mismo permiso dos veces, retorna 200 sin duplicar.

### Criterios de Aceptación
- [ ] authorization-service expone POST /api/permissions/users/{userId} (uso interno).
- [ ] Valida que el usuario exista antes de asignar permisos.
- [ ] Valida que el permiso sea válido (REPORT_READ, REPORT_DOWNLOAD, etc.).
- [ ] Si el permiso ya está asignado, retorna HTTP 200 (idempotencia).
- [ ] Si el usuario no existe, retorna HTTP 404 (causa rollback en auth).
- [ ] Si el permiso es inválido, retorna HTTP 400 (causa rollback en auth).
- [ ] authentication-service invoca este endpoint vía Feign Client en el registro.
- [ ] La transacción en authentication-service es atómica: éxito total o rollback completo.

### Casos Límite y Validaciones
- Caso límite: usuario no existe; retornar 404.
- Caso límite: permiso inválido; retornar 400.
- Caso límite: permiso ya asignado; retornar 200 (idempotencia).
- Caso límite: concurrencia; múltiples asignaciones simultáneas del mismo permiso.
- Validación: ID del usuario debe ser numérico válido.
- Validación: permiso debe estar en lista blanca de permisos conocidos.

### Requisitos No Funcionales Relevantes
- Seguridad: validar que el solicitante tiene autorización para asignar permisos.
- Rendimiento: asignación rápida (<50ms).
- Confiabilidad: operación idempotente.
- Observabilidad: registrar asignaciones de permisos (sin datos sensibles).
- Documentación API: configurar OpenAPI/Swagger y marcar este contrato como interno para evitar exponerlo como API pública.

### Requisitos de Testing (alto nivel)
- Pruebas unitarias: validación de permisos válidos/inválidos, idempotencia.
- Pruebas de integración: persistencia de asignación, consulta posterior de permisos.
- Pruebas concurrentes: múltiples asignaciones simultáneas.

### Consideraciones Técnicas
- Backend / API:
  - `POST /api/permissions/users/{userId}`
  - PermissionController, PermissionService, PermissionRepository
  - Body: `{"permission": "REPORT_READ"}`
  - Respuesta: `{"userId": 1, "permission": "REPORT_READ", "status": "ASSIGNED"}`
- Integración:
  - Endpoint llamado desde authentication-service vía Feign Client.
  - Debe estar dentro de la transacción del registro de usuario en authentication-service.
- Errores:
  - HTTP 400 si permiso inválido.
  - HTTP 404 si usuario no existe.
  - HTTP 409 si hay conflicto (opcional, o 200 si es idempotente).

Objetivo técnico:
Exponer endpoint en authorization-service que asigne permisos a usuarios, permitiendo que authentication-service configure permisos predeterminados al registrar nuevos usuarios.

Alcance funcional:
- Crear endpoint POST /api/permissions/users/{userId}.
- Validar que el usuario exista.
- Validar que el permiso sea válido.
- Asignar el permiso a la relación user_permissions.
- Retornar confirmación de asignación.
- Ser idempotente (no duplicar si ya existe).

Buenas prácticas esperadas:
- Usar DTO para request y response.
- Usar validación de entrada.
- Usar transacciones.
- Ser idempotente.
- Loguear asignaciones sin datos sensibles.
- Documentar endpoint con OpenAPI.
- Aclarar en la documentación que este endpoint se consume solo por Feign desde authentication-service.

Endpoint:
POST /api/permissions/users/{userId}

Headers:
Content-Type: application/json
X-Service-Auth: <token-interno> (opcional, si se asegura con token)

Ejemplo de body:
{
  "permission": "REPORT_READ"
}

Ejemplo de salida exitosa HTTP 200 (asignado o ya existía):
{
  "userId": 1,
  "permission": "REPORT_READ",
  "status": "ASSIGNED",
  "timestamp": "2026-05-13T14:35:00"
}

Ejemplo de permiso inválido HTTP 400:
{
  "timestamp": "2026-05-13T14:36:00",
  "status": 400,
  "error": "Bad Request",
  "message": "Permiso inválido",
  "details": "Permiso 'INVALID_PERM' no es válido. Permisos conocidos: REPORT_READ, REPORT_DOWNLOAD",
  "path": "/api/permissions/users/1"
}

Ejemplo de usuario no encontrado HTTP 404:
{
  "timestamp": "2026-05-13T14:37:00",
  "status": 404,
  "error": "Not Found",
  "message": "Usuario no encontrado",
  "path": "/api/permissions/users/999"
}

Reglas de negocio:
- Solo servicios internos pueden asignar permisos.
- La asignación debe ser idempotente.
- Permisos válidos: REPORT_READ, REPORT_DOWNLOAD, y otros a definir.
- Un usuario puede tener múltiples permisos.
- La asignación es inmediata y persistente.

Criterios de aceptación:
1. Dado un usuario existente y un permiso válido, cuando invoque POST /api/permissions/users/{id}, entonces debe retornar HTTP 200 con confirmación de asignación.
2. Dado un usuario que ya tiene el permiso, cuando invoque el endpoint, entonces debe retornar HTTP 200 sin crear duplicado (idempotencia).
3. Dado un usuario inexistente, cuando invoque POST /api/permissions/users/{id}, entonces debe retornar HTTP 404.
4. Dado un permiso inválido, cuando invoque POST /api/permissions/users/{id}, entonces debe retornar HTTP 400 con detalle.
5. La operación debe ser transaccional y atómica.

Sugerencia de estructura técnica:
- PermissionController
- PermissionService
- PermissionRepository / UserPermissionRepository
- UserPermissionEntity
- PermissionAssignmentDTO
- AssignmentResponseDTO
- GlobalExceptionHandler
